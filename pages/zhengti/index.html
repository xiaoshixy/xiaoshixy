SOW——>业务需求、产品范围描述、战略计划

🔸成功标准、退出标准

# 制定项目管理计划

概念

定义、准备和协调所有子计划 整合为一份综合的项目管理计划

# 输入

点击查看

其他过程的输出

# 工具

  1. 传达项目目标，获得团队对项目的承诺
  2. 获得主要干系人对项目管理计划的一致认可
  3. 阐明每个干系人的角色与职责

# 输出

  • 13个子计划
点击查看

范围，需求，进度，成本，质量，人力资源，沟通，风险，采购，干系人

变更，过程改进，配置

警告

可概括、可详细 ，通过不断更新来渐进明细 。

一旦被确定为基准,则只能通过提出变更请求,并经实施整体变更控制过程批准后,才能 变更。

  • 3大基准
点击查看

范围基准，进度基准，成本基准

  • ⭐项目管理计划内容
点击查看

🔸选择的各个项目管理过程、过程的实施水平、过程的工具与技术、过程之间的依赖关系和相互作用

🔸监控变更、配置管理、实施效果测量基准的方式、方法

🔸选定的项目生命周期和项目阶段、关键审查、干系人之间的沟通需要与技术

# 指导与管理项目工作

概念

领导和执行项目管理计划中所确定的工作，并实施已批准的变更

# 输入

点击查看

项目管理计划——>干计划的事情

批准的变更请求——>干批准的变更

# 工具

点击查看

信息管理系统（PMIS）——>事业环境因素

会议

如何开好会议？

点击查看
  1. 事先制定一个例会制度、明确会议规则
  2. 放弃可开可不开的会议
  3. 明确会议目的和期望结果，明确参会者角色
  4. 发布会议通知
  5. 会议之前将会议资料发给参会人员
  6. 面对面会议，可借助视频设备
  7. 会后及时总结、达成共识
  8. 做好会议纪要
  9. 做好会议后勤保障
  10. 不要把各种会议类型混合在一起

点击查看

🔸可交付成果

🔸工作绩效数据

执行过程中第一手数据

🔸变更请求

纠正措施纠偏差、预防措施防风险、缺陷补救补质量、更新通常改基准

# 变更

# 分类

  • 不同审批权限
点击查看

重大，重要，一般

  • 不同处理流程
点击查看

紧急，非紧急

# CCB

负责 审议、批准、推迟或否决项目变更 ，以及 记录和传达变更处理决定 。

CCB是决策机构 ，不是作业机构。 可以多方人员共同组成，也 可以只一个人 组成。

CCB不是必须要有 ，也不是所有的变更请求都必须要由 CCB 来审批

# 变更请求批准人的选择顺序

  • 首选

项目管理计划或组织流程中指定的责任人。

  • 次选

CCB

  • 再选

“PMO ”、“发起人”、“项目经理”等

# ⭐变更8步骤

点击查看
  1. 提出与记录（PM或CMO收集记录）
  2. 初审变更
  3. 评估（PM 带领团队和相关干系人评估）
  4. 提交CCB审查
  5. 更新、通知、组织实施 （PM 更新、通知，组织团队实施）
  6. 变更实施的监控 （PM 监控基准，监理或 CCB 监控主要成果、里程碑）
  7. 变更效果的评估 （团队验证变更是否正确，客户验证是否满足最初要求）
  8. 判断变更后项目是否纳入正轨 （PM 判断是否纳入正轨）
